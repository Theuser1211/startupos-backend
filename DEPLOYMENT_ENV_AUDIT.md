## 4. Variables Required for Railway Deployment

Railway auto-generates `DATABASE_URL` and `REDIS_URL`. Manual setup needed for:

```env
# Auto-generated by Railway
DATABASE_URL=postgresql://postgres:xxx@containers-us-west-xxx.railway.app:5432/railway
REDIS_URL=redis://default:xxx@containers-us-west-xxx.railway.app:6379

# Must set manually
NODE_ENV=production
PORT=3000
HOST=0.0.0.0
JWT_SECRET=<generate with: openssl rand -base64 48>
GROQ_API_KEY=<from console.groq.com>
VERCEL_TOKEN=<from vercel.com/account/tokens>
LOG_LEVEL=info
```

## 5. Variables Required for Supabase Production Database

From Supabase Dashboard → Settings → Database:

```env
DATABASE_URL=postgresql://postgres.PROJECT_REF:PASSWORD@aws-0-REGION.pooler.supabase.com:6543/postgres
```

Notes:
- Use **Pooler** connection string (port 6543), not direct (5432)
- Use **Transaction** mode for Prisma (connection pooling)
- The `?pgbouncer=true` parameter may be needed for some ORMs
- Service role key is in Settings → API → `service_role` key

## 6. Variables Required for Upstash Redis

From Upstash Console → Database → Redis URL:

```env
REDIS_URL=rediss://default:XXXX@YOUR_REGION.upstash.io:6379
```

Or extract host/port:

```env
REDIS_HOST=YOUR_REGION.upstash.io
REDIS_PORT=6379
```

Notes:
- `rediss://` prefix = TLS enabled (required for Upstash)
- The code now supports `REDIS_URL` directly (added this sprint)
- BullMQ requires Redis 6+ (Upstash is Redis 7 compatible)

## 7. Variables Required for Vercel Deployment

From Vercel Dashboard → Settings → Tokens:

```env
VERCEL_TOKEN=vcp_xxxxxxxxxxxxxxxxxxxx
```

Notes:
- Token needs `deployments:write` scope
- Deployments create a Vercel project automatically
- After first deploy, disable "Deployment Protection" in project settings for public access

## 8. Security Concerns

| Issue | Severity | Location | Recommendation |
|-------|----------|----------|----------------|
| `JWT_SECRET` in .env with weak value | 🔴 Critical | `.env` | Generate with `openssl rand -base64 48`. Never commit real secret. |
| `SUPABASE_SERVICE_KEY` exposed in .env | 🔴 Critical | `.env` | Service role key bypasses RLS. Rotate immediately if committed to git. |
| `GROQ_API_KEY` exposed in .env | 🟡 Medium | `.env` | Rotate if committed to git. Free tier key has low blast radius. |
| `VERCEL_TOKEN` exposed in .env | 🟡 Medium | `.env` | Can create/delete deployments. Rotate if committed. |
| `DATABASE_URL` with password in .env | 🔴 Critical | `.env` | Never commit. Use secrets manager in production. |
| `.env` file in git repo | 🔴 Critical | `.gitignore` | Verify `.env` is in `.gitignore`. |
| No rate limiting on auth endpoints | 🟡 Medium | `server.ts` | Only global rate limit (100/min). Add per-route limits. |
| No CORS restriction | 🟡 Medium | `server.ts:26` | `origin: true` allows any origin. Set explicit origins in production. |
| Prisma `db push` in production | 🟡 Medium | `package.json` | Use `prisma migrate deploy` for production. `db push` can lose data. |

## 9. `.gitignore` Verification

The `.env` file MUST be in `.gitignore`. Current `.gitignore` should contain:

```
.env
.env.local
.env.*.local
```

If `.env` is tracked in git, run:
```bash
git rm --cached .env
git commit -m "Remove .env from tracking"
```

## 10. Production Deployment Checklist

```
[ ] DATABASE_URL set to managed PostgreSQL (not localhost)
[ ] REDIS_URL set to managed Redis (not localhost)
[ ] JWT_SECRET is 32+ chars, randomly generated
[ ] At least one AI provider key configured
[ ] VERCEL_TOKEN set for deployment
[ ] NODE_ENV=production
[ ] LOG_LEVEL=info
[ ] .env not committed to git
[ ] CORS restricted to actual domains
[ ] Prisma migrations used (not db push)
[ ] Supabase service role key rotated if leaked
```
